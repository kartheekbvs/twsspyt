Exception Handling

Exceptions are runtime errors that disrupt normal program flow. Python uses a try/except/else/finally structure to handle them gracefully. This chapter covers the exception hierarchy, all built-in exceptions, custom exceptions, context managers, and best practices.

● Intermediate 📖 Python Programming (2024)

1 · try / except / else / finally

Python · Full try/except Structure 
try:
    # Code that might raise an exception
    result = 10 / 0
except ZeroDivisionError as e:
    # Runs if ZeroDivisionError occurs
    print(f"Caught: {e}")      # Caught: division by zero
else:
    # Runs ONLY if no exception was raised
    print(f"Result: {result}")
finally:
    # ALWAYS runs — cleanup code goes here
    print("Cleanup done")

# ── When no exception ────────────────────────────────────
try:
    result = 10 / 2
except ZeroDivisionError:
    print("Error!")
else:
    print(f"Success: {result}")  # This runs
finally:
    print("Always runs")
▶ Output
Caught: division by zero
Cleanup done
Success: 5.0
Always runs
🔁 Clause Execution Rules

try executes first → if exception: except runs → if no exception: else runs → finally always runs regardless (even with return inside try, or SystemExit).

3 · Handling Multiple Exceptions

Python · Multiple except Clauses 
def safe_divide(a, b):
    try:
        result = a / b
        return round(result, 2)
    except ZeroDivisionError:
        print("Error: Cannot divide by zero")
        return None
    except TypeError as e:
        print(f"Type error: {e}")
        return None

print(safe_divide(10, 2))    # 5.0
print(safe_divide(10, 0))    # Error: Cannot divide by zero → None
print(safe_divide(10, "a"))  # Type error: ... → None

# Multiple exceptions in one except clause
try:
    value = int(input("Enter a number: "))
    result = 100 / value
except (ValueError, ZeroDivisionError) as e:
    print(f"Input error: {e}")

# Bare except catches ALL exceptions (not recommended)
try:
    risky_operation()
except Exception as e:  # Better: catch Exception base class
    print(f"Unexpected: {type(e).__name__}: {e}")
▶ Output
5.0
Error: Cannot divide by zero
None
Type error: unsupported operand type(s) for /: 'int' and 'str'
None

4 · Raising Exceptions

Python · raise & re-raise 
# raise: manually trigger an exception
def set_age(age):
    if not isinstance(age, int):
        raise TypeError(f"Age must be int, got {type(age).__name__}")
    if age < 0 or age > 150:
        raise ValueError(f"Age {age} is out of valid range [0, 150]")
    return age

try:
    set_age(-5)
except ValueError as e:
    print(f"Caught: {e}")

# re-raise: re-raise current exception for caller to handle
def load_config(path):
    try:
        with open(path) as f:
            return f.read()
    except FileNotFoundError:
        print(f"Config file not found: {path}")
        raise   # re-raise same exception

# Exception chaining with 'from'
try:
    raise ValueError("Invalid input") from None  # suppress original
except ValueError as e:
    print(f"Raised: {e}")
▶ Output
Caught: Age -5 is out of valid range [0, 150]
Raised: Invalid input

5 · Custom Exceptions

Python · Custom Exception Classes 
# Custom exceptions inherit from Exception
class AppError(Exception):
    """Base class for application errors."""
    pass

class ValidationError(AppError):
    def __init__(self, field: str, message: str):
        self.field   = field
        self.message = message
        super().__init__(f"Field '{field}': {message}")

class DatabaseError(AppError):
    """Raised on database failures."""
    pass

# Use them
def validate_email(email):
    if "@" not in email:
        raise ValidationError("email", "Must contain '@' symbol")

try:
    validate_email("notanemail")
except ValidationError as e:
    print(f"Validation failed: {e}")
    print(f"Field: {e.field}")
except AppError as e:
    print(f"App error: {e}")
▶ Output
Validation failed: Field 'email': Must contain '@' symbol
Field: email
💡
Custom Exception Hierarchy

Create a base AppError class, then specific subclasses. This allows callers to catch specific errors or all app errors in one except AppError clause — following the Open/Closed principle.

6 · Best Practices

⚠️
Never do bare except:

Using except: without specifying an exception type catches everything, including KeyboardInterrupt (Ctrl+C) and SystemExit. Always specify the exception type(s).

Python · Best Practice Patterns 
# ✅ GOOD: Be specific, catch what you can handle
try:
    data = int(user_input)
except ValueError:
    data = 0

# ❌ BAD: Catch-all silences real bugs
try:
    data = int(user_input)
except:
    data = 0  # Hides ALL errors, even MemoryError!

# ✅ GOOD: Use finally for cleanup, or context managers
with open("file.txt") as f:
    data = f.read()   # File closes automatically, even on error

# ✅ GOOD: Log and re-raise in libraries
import logging

try:
    result = risky_operation()
except Exception as e:
    logging.error(f"Operation failed: {e}", exc_info=True)
    raise   # Let caller decide how to handle it
